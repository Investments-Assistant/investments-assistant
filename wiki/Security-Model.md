# Security Model

The investment assistant is designed to be **unreachable from the public internet**.
The only port visible to the outside world is **UDP 51820** (WireGuard). Everything else
is protected by four independent layers.

---

## Layer 1: WireGuard VPN

WireGuard is built into the Linux kernel (since 5.6). It uses **Noise protocol** with
Curve25519 key exchange and ChaCha20-Poly1305 encryption.

**Key security property**: WireGuard silently discards all packets from unknown peers.
There is **no response, no banner, no error** ‚Äî to a network scanner, the port appears
closed. An attacker cannot even confirm WireGuard is running. Compare this to an SSH
server which at minimum responds with its version banner.

Authentication is based entirely on **Curve25519 public keys** ‚Äî there are no passwords,
no login forms, no brute-force vectors.

**Pre-shared key (PSK)**: on top of the asymmetric key exchange, an optional 256-bit
symmetric PSK is mixed into the session key derivation. This provides **post-quantum
security**: even if Curve25519 is broken in the future (quantum computers), an attacker
would also need the PSK to decrypt historical traffic.

**Network topology**:
```
Your phone / laptop (WireGuard client, 10.8.0.x)
    ‚îÇ  UDP 51820 ‚Äî only port forwarded on router
    ‚ñº
Router ‚Üí Pi 5 (WireGuard server, 10.8.0.1)
    ‚Üí HTTPS 443 ‚Üí Investment Assistant
```

The Pi has a **static VPN IP** of `10.8.0.1`. Clients are assigned IPs in `10.8.0.0/24`.
The `ALLOWED_IPS` setting defaults to `10.8.0.0/24` ‚Äî only VPN peers can reach the app.

## Layer 2: DOCKER-USER iptables chain

Docker modifies iptables rules to expose container ports to `0.0.0.0` on the host. If
Nginx binds to `0.0.0.0:443`, Docker's default rules make port 443 reachable from any
IP that can reach the Pi ‚Äî including the public internet (if the router forwards port 443,
which it should not, but misconfigurations happen).

`scripts/setup.sh` adds rules to the `DOCKER-USER` chain that drop all traffic to ports
80 and 443 except from the VPN subnet and LAN:

```bash
# Drop 80/443 unless from VPN or LAN
iptables -I DOCKER-USER -p tcp --dport 80 -j DROP
iptables -I DOCKER-USER -p tcp --dport 443 -j DROP
iptables -I DOCKER-USER -p tcp --dport 80 -s 10.8.0.0/24 -j ACCEPT
iptables -I DOCKER-USER -p tcp --dport 443 -s 10.8.0.0/24 -j ACCEPT
iptables -I DOCKER-USER -p tcp --dport 443 -s 192.168.0.0/16 -j ACCEPT
```

These rules operate at the **kernel netfilter level** ‚Äî before any application code
runs. Even if Nginx is misconfigured or has a vulnerability, the kernel silently drops
the packet.

`iptables-persistent` is installed to survive reboots.

## Layer 3: Nginx IP whitelist + TLS + rate limiting

```nginx
set $allowed 0;
if ($remote_addr ~* ^10\.8\.0\.)    { set $allowed 1; }  # WireGuard VPN
if ($remote_addr ~* ^192\.168\.)    { set $allowed 1; }  # LAN
if ($remote_addr ~* ^172\.(1[6-9]|2[0-9]|3[01])\.) { set $allowed 1; }  # Docker bridge
if ($allowed = 0) { return 403; }
```

This is a **redundant** application-layer check. If the iptables rules are ever removed
or misconfigured, Nginx will still reject non-whitelisted IPs.

The Docker bridge range (`172.16‚Äì31.x.x`) is whitelisted to allow health checks from
within Docker.

**TLS configuration**:
- Self-signed certificate generated by `make gen-certs` (4096-bit RSA, 10-year validity)
- Protocols: TLS 1.2 and 1.3 only (TLS 1.0/1.1 are disabled)
- Ciphers: ECDHE with AES-256-GCM or ChaCha20-Poly1305 only (no RSA key exchange,
  no CBC mode)
- `ssl_session_tickets off`: disables TLS session tickets, preventing ticket key
  compromise from decrypting past sessions (forward secrecy)
- `Strict-Transport-Security: max-age=31536000; includeSubDomains`: browsers remember
  to always use HTTPS for this origin

**Security headers**:
- `X-Frame-Options: DENY` ‚Äî prevents clickjacking
- `X-Content-Type-Options: nosniff` ‚Äî prevents MIME-type sniffing attacks
- `Referrer-Policy: strict-origin-when-cross-origin` ‚Äî limits referer leakage
- `Content-Security-Policy: default-src 'self'; connect-src 'self' wss:;` ‚Äî blocks
  loading external scripts or making cross-origin requests (except WebSocket to same host)

**Rate limiting** (configured in `config/nginx/rate-limit.conf`):
- `limit_req_zone`: limits request rate per IP
- `limit_conn_zone`: limits concurrent WebSocket connections per IP (`ws_conn 3`)

## Layer 4: FastAPI IP check

```python
def require_allowed_ip(request: Request) -> None:
    ip = _get_client_ip(request)
    if not settings.is_development and not settings.is_ip_allowed(ip):
        raise HTTPException(status_code=403, detail="Access denied")
```

This is the innermost defence ‚Äî a Python function that runs as a FastAPI dependency
on every request. It parses `X-Forwarded-For` (set by Nginx's
`proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for`) to get the real client IP.

`settings.is_ip_allowed()` uses Python's `ipaddress.ip_address` / `ip_network` to
check if the IP falls within any configured CIDR block. This supports both IPv4 and IPv6.

For WebSocket connections, the IP check is replicated directly in the `websocket_chat`
handler (FastAPI's `Depends` doesn't apply cleanly to WebSocket handlers):

```python
@router.websocket("/ws/chat/{session_id}")
async def websocket_chat(websocket, session_id):
    ip = _get_client_ip(websocket)
    if not settings.is_development and not settings.is_ip_allowed(ip):
        await websocket.close(code=4003, reason="Access denied")
        return
```

In **development mode** (`ENVIRONMENT=development`), the IP check is skipped ‚Äî making
it easy to run the app locally without setting up VPN. The FastAPI `/docs` and
`/openapi.json` endpoints are also enabled only in development mode.

## Secret management

All secrets (API keys, database passwords) are stored in `.env` which is:
- Listed in `.gitignore` ‚Äî never committed to git
- Mounted into the app container via `env_file: .env` in `docker-compose.yml`
- Provided to GitHub Actions via **environment secrets** in the `dev` and `prod`
  environments (managed by Terraform in `core-infra`)

**Database password**: `POSTGRES_PASSWORD` is required (not defaulted to a known string).
Docker Compose enforces this with:
```yaml
POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:?Set POSTGRES_PASSWORD in .env}
```
If `POSTGRES_PASSWORD` is missing, `docker compose up` fails with a clear error.

## SonarCloud / secret scanning

GitHub's **secret scanning** and **push protection** are enabled on both repositories
via Terraform:
```hcl
security_and_analysis {
  secret_scanning { status = "enabled" }
  secret_scanning_push_protection { status = "enabled" }
}
```

Push protection blocks a `git push` if a secret pattern (API key, private key, etc.) is
detected in the diff. SonarCloud's SAST scan in the CI pipeline also flags hardcoded
credentials.

## Pi-hole isolation

Pi-hole binds to port 53 (DNS) with `cap_add: - NET_ADMIN`. This capability allows it to
manage network interfaces for DNS, but it's scoped only to the Pi-hole container ‚Äî the
app container has no extra capabilities.

Pi-hole's admin web UI is exposed on `8080:80` ‚Äî intentionally not proxied through Nginx
and not TLS-protected, because it's only accessible on the LAN (firewall blocks it from
the internet).
